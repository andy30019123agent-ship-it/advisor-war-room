## 🔴 1. FINMIND_TOKEN 從沒被設成 GitHub Secret——明天其實跑在免費 300 次/hr，不是 600

`gh secret list` 只有兩個：`TG_BOT_TOKEN`、`TG_CHAT_ID`。**沒有 `FINMIND_TOKEN`。**

`daily-update.yml`／`alerts.yml` 兩支 workflow 都寫了 `env: FINMIND_TOKEN: ${{ secrets.FINMIND_TOKEN }}`，但既然這個 secret 不存在，GitHub Actions 會把它解析成空字串——07-18 唯一一次真實執行的 log 白紙黑字印出：
```
FINMIND_TOKEN: 
```
`warroom/finmind_cache.py` 的 `get_loader()` 邏輯是 `if token: dl.login_by_token(...)`，空字串走不進這行，代表**整條管線一直都在用 FinMind 免費匿名額度（300 次/hr），從來不是程式註解講的「有 token 600/hr」**。（log 裡出現的 `login_by_token: Login success` 是 FinMind SDK 自己內部匿名初始化的訊息，跟我們的 token 無關，不能當作已登入付費/免費 token 的證據。）

這件事直接放大下一條的風險：明天 14:30 這輪估算要打 ~250 次 FinMind（見第 2 節估算表），對 300/hr 的免費額度已經吃掉 83%，安全邊界很薄。

**怎麼修**：去 FinMind 官網申請免費 token（不用付費，只是要有帳號登入才能拿到 300→600 的額度倍增），貼進 repo 的 GitHub Secrets（Settings → Secrets and variables → Actions → New repository secret，名稱 `FINMIND_TOKEN`）。這是本次唯一建議「今晚就做」的動作，5 分鐘搞定，直接把明天的額度風險腰斬。

## 🔴 3. FinMind 單次呼叫在 CI 環境實測出現 5 分鐘延遲，30 分鐘 timeout 沒有想像中寬裕

07-18 唯一一次真跑的 log，`taiwan_stock_institutional_investors_total`（大盤外資合計）這一次呼叫，發出請求到收到下一行 log 之間：
```
06:41:25.6426058Z  download Dataset.TaiwanStockTotalInstitutionalInvestors, data_id:
06:46:39.2305121Z  login_by_token: Login success   ← 下一個動作
```
**間隔 5 分 14 秒**，是官方 log 時間戳的實測數字，不是推測。那次整個 job 只打了約 12 次 FinMind（2 檔個股×8 + TAIEX/TPEx + 外資合計），扣掉這一次異常延遲，其餘呼叫都在秒級完成——代表 FinMind 端本身有機率出現偶發的長延遲（不確定是真的慢還是被限流重試，但現象是實測到的）。

明天的新管線估算要打 ~250 次（見下表），如果其中哪怕 3-4 次撞到類似的分鐘級延遲，單獨就可能吃掉 15-20 分鐘，加上其餘呼叫的正常耗時，**有實質機率逼近甚至超過 `daily-update.yml` 的 `timeout-minutes: 30`**。一旦 timeout 到底，job 被硬殺，當天 `data/`／`public/data/` 完全不會更新（tracked 個股、picks、大盤全部開天窗），且不會有任何降級或部分結果——是全有全無的失敗模式。

**怎麼修**：
1. 今晚先把第 1 條的 token 補上（降低額度逼近風險，順帶把 hr 額度倍增到 600，讓 429/重試的機率下降，間接降低延遲風險）。
2. 建議把 `timeout-minutes` 從 30 調到 45（成本幾乎是 0，public repo Actions 分鐘數全免費），給這次「第一次實戰」多一點緩衝，先求穩過關，之後再視實測結果調回。
3. 沒有本機重跑 build_snapshots 對真實網路計時（刻意迴避——這樣做會現場真的消耗掉 FinMind 額度，等於在檢查階段就先吃掉明天的配額，不划算），時間評估改用上面「CI 真實 log 實測延遲＋呼叫數估算表」佐證，這點跟 Andy 說清楚：**這次沒有「本機跑兩次計時」的直接證據，是用已知的唯一真實樣本＋推算**，如果想要更硬的證據，需要接受消耗一次額度、明天之前手動 `workflow_dispatch` 跑一次觀察真實耗時。

## 🟡 4. picks 評分「半殘名單」風險：不會當機，但會有排序偏誤且不可見

`picks.py._fetch_light._grab()` 對每個候選股的每個 dataset 呼叫都用 `except Exception: return None` 全部吞掉，任何失敗（含額度用完）都變成「這項指標缺值，不給分」，是善意的防呆設計。

但額度用完是**系統性**失敗——同一小時內額度歸零後，接下來全部呼叫都會失敗。候選池組裝順序是 opportunities → tracked → universe（`build_candidate_pool`），如果額度剛好在池子跑到一半時用完，**排在後段的候選股會全部拿到空指標、系統性算不到分**，不是隨機缺資料，是「越晚評分越吃虧」的結構性偏誤。不會報錯、不會有警訊，因為每一檔看起來都只是走了正常的「資料不足不給分」路徑，沒有任何地方會記錄「這是額度問題，不是這支股票真的資料缺」。

被選新股的 `analyze()`（上限 6 檔）額外有 `FinMindUnavailable` 顯式拋出機制，但 `generate_picks()` 呼叫它的迴圈一樣用 `except Exception: pass` 全部吞掉，該股直接從白名單消失、不進 picks，也沒有任何記錄留下「這檔其實達標但因額度被跳過」。

**這次不建議马上改**（牽動評分/白名單邏輯，屬於功能行為變更，不是純維運修復），但建議記一筆技術債：至少把 `counter[0]`／`analyze_calls` 這兩個既有的呼叫數統計（`build_snapshots.main()` 已經有印出 `FinMind 呼叫估 X 次`）順手加一行「若接近或觸頂額度上限，在 daily.picks.note 附註一句降級說明」，讓使用者知道今天的精選名單可能不完整，而不是誤以為「就是沒有更好的股票」。

## 🟡 5. `fetch_market()` 外資合計端點繞過快取，且正是實測出現 5 分鐘延遲那個端點

`warroom/market.py` L50 `dl.taiwan_stock_institutional_investors_total(start_date="2026-07-01")` 直接呼叫 `DataLoader` 方法，沒有走同檔案其他呼叫都用的 `cached_fetch()`。`fetch_market()` 在整條管線裡被呼叫兩次（`warroom.update` 一次、`build_snapshots.fetch_market_inputs()` 內又呼叫一次拿 `foreign_net_yi`），代表這個端點**明天會被真實打兩次**，而且巧合的是這正是唯一一次真跑實測出現 5 分鐘延遲的那個端點（第 3 條）。

**怎麼修**（不急，但值得順手做）：把這行改成走 `cached_fetch("taiwan_stock_institutional_investors_total", loader=dl, start_date="2026-07-01")`，跟同檔其他呼叫一致，兩次呼叫變一次快取命中，少打一次這個目前唯一有實測延遲證據的慢端點。

## 🟢 已驗證沒問題

### `/api/quote` 限流機制：本機 mock 網路實測 125 次請求，行為完全符合設計
用 `unittest.mock` 風格手動 monkeypatch `is_trading_window()`＝True（避開非交易時段的整批 stale 短路）＋假造 `_fetch_mis`（不打真網路），本地跑 125 次不同 `ids` 的請求：
```
125 次不同 ids 請求：成功 120，被擋 5
第一次被擋＝第 121 個請求（預期第 121 個，上限 120/hr）
限流檔最終狀態：{'bucket': '2026-07-19T08', 'count': 120}
上限已滿時再查詢新 ids：429(符合預期，上限已滿)
```
120/hr 上限精準卡在第 121 次，計數器打滿後不再累加（沒有溢位風險），快取命中／額度已滿兩條路徑都符合程式碼設計。錯誤處理完整：`do_GET` 把所有未預期例外包成 503 JSON，不會讓前端拿到裸 500 白屏。（測試殘留的 `/tmp/quote_rate_limit.json` 已在腳本內清乾淨，不影響這台機器之後的真實使用。）

MIS 上游對高頻的真實容忍度（多 instance 併發時的整體流量）無法在不動用真實流量的前提下驗證，這點程式碼自己的 docstring 已經誠實揭露侷限（「不是全域限流，冷啟或換 instance 會歸零」），非 bug，是已知設計取捨。
