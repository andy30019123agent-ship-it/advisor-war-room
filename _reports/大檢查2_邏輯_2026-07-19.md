## 🔴 R1 — 中長線判讀「翻多條件」標籤寫死，偏多股實為翻空觸發（語意相反）

**重現（2330 實值）：** swing bias =「中性偏多」，`flip_condition` =「跌破 防守價 2,107 且連 2 日賣超 → 轉中性偏空」。畫面 `MidLongReads.tsx` 把 label 寫死成「翻多條件」，使用者看到：

> 翻多條件：跌破 防守價 2,107 且連 2 日賣超 → 轉中性偏空

label 說「怎麼翻多」，內文卻描述「怎麼翻空」，方向完全相反。

**根因：**
- `warroom/primary_decision.py:635` docstring 明講「flip_condition 一律指向**相反方向**的下一個 stance」。`_direction_read`（`:646-655`）：lean=up → flip 是「→轉中性偏空」；lean=down/neutral → flip 是「→轉中性偏多」。
- `app/src/components/MidLongReads.tsx:24` 硬寫 `<span>翻多條件</span>`，不看方向。

**影響面：** 所有 bias=偏多/中性偏多 的股票（2330、2382、2912、2207…swing 或 mid）。偏多持有者可能把「停損/降評觸發」誤讀成「加碼/翻多訊號」，做出反向操作。

**修法：** label 依 bias lean 動態化 — up → 顯示「翻空條件」；down/neutral → 「翻多條件」。或直接解析 `flip_condition` 尾端「→轉XX」字樣決定 label。

## 🔴 R2 — roster rank_move 跨艙比較，換池股顯示假的名次箭頭

**重現：** 某股昨天在 `on_deck rank 2`，今天在 `research rank 1` → `1 < 2` → 顯示「↑」（看起來像名次躍升），但兩個名次分屬不同池、不可比。

**根因：** `warroom/picks.py` `_rank_move`（`:696-706`）只讀 `prev.get("rank")`，不比對 `pool`；但 docstring 寫「與上一份 roster **同池**名次比」。上一筆其實有存 `pool`（`:787`）卻沒被用。

**修法：** `prev.get("pool") != 當前 pool` 時回傳「−」（或標「換艙」），不給箭頭。

## 🟡 前排（依嚴重度）

### Y1 — quote overlay：盤中即時價與收盤劇本混用 →「劇本說沒破、即時價已破」同框
- `Today.tsx:200-206`「距防守 %」與紅點緊張度用 **live 即時價**（`currentPrice = live ? quote.price : close`）。
- `ScenarioCards.tsx:78` 劇本 trigger 的距離 % 用 **snapshot 收盤**（`(price - close)/close`）；劇本 invalidation 文字寫「**收盤**跌破防守價…失效」。
- 盤中若 live < 防守價：上方顯示「距防守 −X%」+紅點（讀作「已破防守」），但劇本卡仍以收盤把「劇本二・守住防守價」列為主劇本、距離%從舊收盤起算 → 同一畫面兩個矛盾訊號。
- **誠實呈現建議：** 距防守/緊張點若用 live，就在破防守時明講「盤中暫破，以**收盤**確認」；或劇本卡的距離% 也吃 live 並在標題標「依 MM-DD 收盤劇本」。兩處價格基準要嘴上講清楚，不要一個用 live 一個用 close 卻都不標。

### Y2 — 金融股研究卡顯示不可比的 PER 分位（掩蓋真正計分的 PBR）
- 計分正確：金融股走 PBR 路徑，PBR=None → 估值貢獻 0，**不回退 PER**、不 crash（`picks.py:476-486`、warning 也 key 在 `pbr_pctile` `:285`）。
- 但研究卡 `_build_reasons` 無條件塞 `per_pctile`，PBR 被排到第 4 條、被 `[:3]`（`picks.py:547`）砍掉。prod 實測 **華南金 2880** 卡片顯示「PER 落在歷史 92% 分位」— docstring 自己說這對金融股「不可比、易矛盾」（`:443`），且根本沒拿來計分。
- **修法：** `_build_reasons` long 分支遇 `is_financial` 改顯示 PBR 分位。

### Y3 — picks 同日冪等在 trade_date=None 時失效（舊 bug 回歸）
- 冪等 guard（`picks.py:715`）需 `data_date` 且等於 roster 存的 date 才不+1。但兩個呼叫端日期來源不同：`generate_picks(data_date=market_inputs["trade_date"])`（`build_snapshots.py:1183`）vs `save_roster(data_date=meta.data_date)`（`:1233`，會 fallback）。
- 正常交易日兩者相等 → 冪等成立（prod roster 2026-07-17，tenure 全=1，同日重跑未+1，✅）。但 **trade_date=None（假日/API 失敗批次）** → generate_picks 拿到 None → guard 停用 → 每重跑一次 tenure+1，而 save_roster 仍寫 fallback date；即 f852a12「同日三跑變3日留任」在缺 trade_date 時重現。
- **修法：** 把 `meta.data_date`（resolved_date）餵給 generate_picks，別用原始 `market_inputs["trade_date"]`。

### Y4 — 輪動席位 sector 字彙不對齊，資金輪動對大半 universe 靜默失效
- tier 靠 `tier_by_id.get(sid) or tier_by_group.get(sector)`（`picks.py:875`）。universe.json 的 sector（工業電腦/汽車/水泥/電子代工/電信/航運…）與 tw_sectors.json 的 group（軍工航太/封裝測試/AI半導體/金融/PCB載板/半導體設備/散熱）幾乎不重疊。
- 結果：多數股 tier=None → `apply_rotation` 只作用於 `"lag"`、on_deck 席只來自 `"lead"` → 整個輪動（領先加席＋落後降權）對它們**靜默不作用**、不 crash、不報錯。重現：工業股全面反攻，研華（工業電腦）該進領先輪動席 → 永遠進不了。
- **修法：** 對齊 universe.sector 與 tw_sectors.group 命名（或加 mapping 表）/ 補齊各 group 的 stock_ids。

### Y5 — 冷靜期連敗判定邊界（journal FIFO）
- `getLossStreak`（`journal.ts:161`）從最新賣單往回數 isLoss===true；遇 **isLoss=null（配不到成本的孤兒賣單）就中斷**。若使用者在連敗中補記一筆「漏配買進」的賣單（例如只記賣沒記買），最新一筆變孤兒 → streak 歸 0 → **冷靜期保護整個消失**。
- 同日多筆依 `created_at`（記錄當下時戳）排序，非真實成交時序：同日「先記賣、後補買」→ 賣單處理時庫存還沒進 queue → 變孤兒 → 中斷連敗。
- **修法：** 孤兒賣單改「跳過不中斷」而非中斷（保守方向反了，寧可續數）；或記一筆時要求同股買賣至少帶成交時間。FIFO 核心數學（部分成交/跨批/孤兒損益）本身正確。

### Y6（補充）— 燈號「連續」與敏感度
- **外資連續＝有資料日的連續，不是日曆交易日的連續**：`chips_v2._streak` 只看傳入序列，缺一個交易日的外資資料是隱形的、不算中斷（實測 [5,8,3]→3，即使中間跳過一天）。中間出現 0（淨額剛好持平）才中斷（[5,0,8,3]→2）；最新日=0 → streak 0。window=10 也把可量測連續數截在 10（prod days=10 疑似截斷）。對 ≥3 門檻影響小，但「連買X日」文案可能跨越資料缺口。
- **技術燈 red 不敏感、green 打不到**：red 需 price<MA20 **且** <MA60；模擬 60 日純盤整（drift 0）全程 yellow、0 次翻燈 → 不是「反覆翻燈」問題。反而 green 需嚴格 price>MA20>MA60>MA120，盤整期幾乎不可能成立 → 燈號結構性偏離 green（偏保守）。且 <120 根資料時 ma120=None → 永遠不 green。
