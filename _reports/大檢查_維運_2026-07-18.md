## 3. 維運風險

### 🔴 重點查證：daily-update.yml 有沒有漏 commit 校正管線的 log 檔？——**查證結果：沒漏，管線設計正確**
這是本次檢查的重點懷疑，逐層查證如下：
1. `forecast_log.json`／`scenario_log.json`／`prob_calibration.json` 三個檔案的寫入邏輯都在 `warroom/build_snapshots.py` 的 `main()`（第 761-786 行）裡**無條件執行**——`write_json(FORECAST_LOG, log)`、`sync_scenario_log(...)`、`sync_calibration()`，路徑都是 `data/forecast_log.json` 等**相對路徑**，不是寫到 `/tmp` 或別的地方。
2. `recommendation_log.json` 寫入邏輯在 `warroom/analyze_tw.py` 的 `analyze()` 尾端（第 393-398 行），`warroom.update` 對每檔追蹤股都會呼叫 `analyze(sid)`，所以每天跑 `python -m warroom.update` 就會落一筆。
3. `daily-update.yml` 的 commit 步驟是 `git add data/ public/data/`——**這是資料夾層級的 add，會遞迴撈進整個 data/ 底下所有變動**，三個 log 檔全部都在 `data/` 底下，所以理論上會一起被撈進去 commit。
4. 拿唯一一次真實跑過的 bot commit（`bf7e8ae`，`gh run list` 查到的 workflow_dispatch 手動觸發，非排程）實測 `git show --stat` 驗證：確實 commit 了 `data/recommendation_log.json`（forecast_log／scenario_log 沒被動到，是因為那次跑的時候這兩支管線的程式碼**還沒寫出來**，是後來才加的 feature，時間序上對得起來，不是 bug 的證據）。

**結論：目前程式碼邏輯上不會白記，`git add data/` 的寫法本來就會連帶撈到新增的 log 檔。** 但要注意：**這個結論還沒被「排程觸發」實測過**——見下一條。

### 🟡 三條 workflow 都還沒被排程真正跑過一次，全部只有手動 workflow_dispatch 紀錄
`gh run list` 查證：`alerts.yml`／`daily-update.yml` 各自只有 1 筆執行紀錄，都是今天（07-18，週六）手動 `workflow_dispatch` 觸發，**不是 cron 排程觸發**。原因不是 bug：這兩支 workflow 的 yml 檔本身也是今天（07-18）才加進 repo 的，而它們的 cron 都是「平日」（`1-5` = Mon-Fri），今天是週六，加上去之後還沒遇到過一個平日，所以排程本來就還沒機會發動——**不是排程壞掉，是還沒到時間**。
`weekly-brief.yml` 也一樣，只等到下週日 20:00（台北）才會第一次被排程觸發。
**這件事本身沒壞，但代表「排程觸發後真的會不會正確 commit／推播」目前完全沒有實測過**，全部驗證都是靠手動 dispatch＋讀程式碼推論。**建議**：下週一（07-20）確認一下 `daily-update` 有沒有在 UTC 06:30（台北 14:30）準時排程跑起來、且 commit 有進去；下週一到週五確認 `alerts` 有沒有在盤中準時發動。這是唯一一件「查了但還沒有真實證據，需要 Andy 之後回頭驗一次」的項目。

### 🟢 三條 workflow push race：實際重疊風險低
`alerts.yml` cron 視窗＝UTC 01:00-05:59（台北 09:00-13:59），`daily-update.yml`＝UTC 06:30（台北 14:30），`weekly-brief.yml`＝UTC 12:00 週日（`alerts`／`daily-update` 只在平日跑，週日完全不會撞）。alerts 視窗跟 daily-update 之間有 31 分鐘空檔，三者排程時間本來就刻意錯開，不會互撞。且各自都設了 `concurrency: group` + `cancel-in-progress: false`，同一支 workflow 自己重疊時是排隊而不是平行跑，設計正確。**唯一殘留風險**：如果哪次 alerts 因為網路問題跑很久（10 分鐘 timeout 到底），有極小機率跟 daily-update 的 06:30 觸發點重疊——機率低、後果也只是其中一個 push 失敗需要重跑，不是資料損毀，不列紅燈。

### 🟢 GitHub Actions 免費額度：**這個顧慮可以完全解除，repo 是 public**
`gh api /repos/.../advisor-war-room` 查證 `"private": false`。**Public repo 的 GitHub Actions 分鐘數完全免費、沒有月配額上限**，原本擔心的「alerts 每 15 分跑一個月會不會爆額度」不成立，不需要精算。
（附計算給參考，即使將來轉 private 也心裡有數：alerts 平日 09:00-13:59 每 15 分＝每個平日 20 次執行，一次最低計費 1 分鐘，20×5×4.33週≈433 分/月；daily-update 平日 1 次、實測跑 6 分多鐘＋計費進位算 7 分＝5×4.33×7≈152 分/月；weekly-brief 每週 1 次、約 1 分鐘計費＝4.33 分/月。合計約 **590 分/月**，遠低於 private repo 的免費 2000 分／月上限，就算轉 private 也夠用。）

### 🟢 Vercel 免費額度：低風險
會觸發 build 的事件＝daily-update（平日 1 次，只在資料真的有變動時才 commit）＋alerts（只在去重狀態變動時才 commit，通常同一天只變一次）＋人為開發 push。目前一天最多看到約 10 次 push（今天因為是密集開發日），Hobby 方案每月 6000 build 分鐘，這個頻率完全用不完，不是風險點。

### 🟢 FinMind 免費額度：目前規模下夠用
`data/tracked_stocks.json` 目前只追蹤 **2 檔**（2330、2454）。有 `FINMIND_TOKEN` 時額度 300→600 次/hr（`finmind_cache.py` 註解證實），且同日同 dataset 有檔案快取不重抓。`alerts.py` 主要走 TWSE 官方即時 API，FinMind 只當 fallback，平日盤中每 15 分最多打一次 fallback、量很小。使用者在 Vercel 上查未追蹤個股（`api/analyze.py`）另外用 `finmind_lite.py` 直連 REST，且有 `/tmp` 當日快取防連點。以目前 2 檔追蹤＋單人使用規模，額度非常寬裕；等追蹤清單擴大到 10+ 檔時可以回頭重估。

## 4. 相依

### 🟡 requirements 沒 pin，GH runner 每次裝最新版有漂移風險
- `pyproject.toml` 只宣告 `pandas`（無版號，且只服務 Vercel serverless 那支 `api.analyze:handler`）。
- 沒有任何 `requirements.txt`。
- `daily-update.yml` 直接 `pip install pandas FinMind yfinance`（無版號釘選）——每次 CI 跑都會裝當下最新版，FinMind／yfinance 這種第三方資料源 SDK 常有 breaking change（改回傳格式、改參數名），一旦上游發新版，排程可能無預警壞掉，且不容易第一時間發現（跑起來不一定報錯，可能是資料悄悄變形）。
- `alerts.yml`／`weekly-brief.yml` 刻意寫成純 stdlib（`urllib` 直打 API），沒有這個問題，設計得不錯。
**怎麼修**：把本機 `.venv` 目前驗證過能動的版本（`pandas==2.3.3`、`FinMind==2.0.4`、`yfinance==1.2.0`、連帶的 `numpy==2.0.2`）凍成 `requirements.txt`，workflow 改 `pip install -r requirements.txt`。之後要升版就是主動決定，而不是被動被 CI 每天偷偷升級。

### 🟡 node 依賴：package.json 全用 `^` 浮動版號，Vercel buildCommand 用 `npm install` 不是 `npm ci`
`app/package-lock.json` 有 tracked（228K），但 `vercel.json` 的 `buildCommand` 寫 `npm install`——沒有嚴格鎖定成 lockfile 那份，跟 `npm ci` 比起來允許在 lockfile 範圍外做 semver 內的浮動安裝。風險比 Python 那邊低（至少有 lockfile 存在），但建議改成 `npm ci` 讓 Vercel 每次 build 都是 100% 可重現、不會因為某個依賴出新版而悄悄變動。

## 5. 災難恢復

### 🟢 data/*.json log 檔壞掉／被覆蓋：有天然的恢復手段，且程式碼本身就有防呆
- 因為每天的 `daily-update.yml` 都會把 `data/` 整包變動 commit 進 git，**每一天的資料快照都在 git 歷史裡**，log 檔壞掉的話直接 `git checkout <上一個好的 commit> -- data/xxx.json` 或 `git revert` 就能復原，不需要額外備份機制。
- 程式碼層面本身也有 fail-closed 設計：`scenario_calibration.py`／`track_record.py` 讀到壞檔（JSON 格式錯、非預期結構）時的處理是「跳過本次寫入＋印警告＋回 None」，**不會因為一次寫入失敗就摧毀既有歷史樣本**（這是看程式碼註解＋邏輯確認的，不是猜測）。
- 唯一沒被涵蓋的是「同一天內、git commit 之前」的中間壞檔——但這種情況本來就还沒推上 git，本機重跑一次 `python -m warroom.update` + `build_snapshots` 就能重算，不算資料遺失。

### 🟡 Vercel 掛掉時使用者看到什麼：沒有離線 fallback，但機率低、後果可控
GitHub Pages 首頁（`index.html`）用 meta-refresh 無條件導去 Vercel 網址，如果 Vercel 剛好掛掉，使用者會直接卡在 Vercel 自己的錯誤頁，**不會自動退回舊版 `reports/` 靜態頁**（雖然那兩個連結技術上還在 HTML 裡，但要使用者手動注意到、且瀏覽器沒把 meta-refresh 跳轉完才看得到，實務上幾乎不可能）。Vercel Hobby 方案本身可用性一般不錯，不是急迫風險，但如果 Andy 在意「萬一 Vercel 掛了怎麼辦」，可以考慮把首頁改成「先顯示卡片清單＋倒數幾秒才跳轉」而不是立即無條件 refresh，這樣至少使用者能看到有備援連結可點。
