## 🔴 會誤導投資決策

### R1. forecast.scenarios 直接吐「被 warning 判定不可信」的估值目標價，且與扇形圖自相矛盾
**現況（live）**：2454 現價 3370，`context.valuation.warning` 已明講「Base 1297.9 與現價偏離 61%，可能低估、不作為減碼依據」。但 `forecast.scenarios` 仍原封不動塞：
```
"scenarios": { "bear": 930.1, "base": 1297.9, "bull": 1331.6 }
```
同一個 forecast 區塊裡，GBM 扇形 m6 的 p10 全程 ≥1534、p50≈2200–2900，**三條「合理價/樂觀目標」線全部躺在扇形雲的最底部之下**。使用者看到「樂觀目標 Bull 1,332」掛在一檔 3,370 的股票上，等於被告知「連最樂觀都要−60%」。
- 根因：`forecast.build_forecast()`（forecast.py L196–197）盲取 `fair_value.bear/base/bull`，**完全沒讀 `valuation.warning`**。primary_decision 花大力氣讓「估值悲觀不得觸發減碼」（§3.3），forecast.scenarios 卻把同一個悲觀原始值重新端到使用者面前，且無任何警示旗標。
- 重現：任何 `valuation.warning` 非 null 的股票（Base 偏離現價 >30%）都會這樣；2454 現在就是。
- 修：forecast.build_forecast 收到 `valuation.warning` 時，`scenarios` 一律帶 `"warning": <該字串>`（或整組回 null），前端據此不畫線或標「模型可能低估，僅參考」。至少要跟 primary 用同一個護欄，不能一邊壓抑一邊裸奔。

## 🟡 潛在（會誤導但有 disclaimer／需特定條件）

### Y1. 劇本機率「不看距離」，與扇形圖的下檔機率正面打臉
`short_scenarios` 的 risk%（跌破防守）純查表（三燈×籌碼＋大盤/籌碼修正），**完全不看防守價離現價多遠**。結果：
- 2330：防守 2107（−8%），risk＝**50%**（劇本一，最顯眼）。但 forecast `week_range_70`＝[2158, 2427]，下緣 2158 **高於**防守 2107 → 扇形說「一週內 <15% 機率碰到 2107」。
- 2454：防守 2876（**−14.6%**），risk 一樣**50%**。扇形 `week_range_70`＝[3023, 3727]，下緣 3023 遠高於 2876；GBM m1 到 d21 的 p25＝2851 才略破防守 → 真實破防機率 ~25%，不是 50%。

兩檔防守距離差一倍（−8% vs −15%），劇本破防機率卻都印 50%。使用者把「劇本一 50% 跌破」跟旁邊「一週 70% 區間站在防守之上」並排看，會被兩套模型的相反訊號搞糊塗。雖有「非統計勝率」disclaimer，但 50% 是整頁最大的數字。
- 修：機率修正項加一個「距離因子」——防守價距現價越遠，risk 基準越低（可用扇形同一套 vol 算「一週內觸及防守的機率」當先驗），或在劇本卡片直接標「需再跌 X% 才觸發」讓 50% 有尺度感。

### Y2. forecast_log 壞檔＝靜默清空歷史，準確度悄悄歸零
`build_snapshots._load_forecast_log()`（L639–646）JSON 壞掉 → `except: return []`，接著 `main()` 無條件 `write_json(FORECAST_LOG, log)`（L777）→ **壞檔被今天這一筆覆寫，過去所有 prob_range 歷史全沒了**。命中率需 ≥10 樣本，一旦被清就永遠停在「樣本累積中」，使用者看到的「準確度累積中」是假的常態，沒有任何錯誤浮現。
- 對比：`scenario_calibration` 與 `track_record` 的 log 都是「壞檔 fail-closed、不覆寫」來保護歷史；只有 forecast_log 反其道而行（程式碼註解說是「刻意不同」，但代價是靜默資料流失）。
- 重現：手動把 `data/forecast_log.json` 改成壞 JSON，跑一次 build → 歷史被清、無警告。
- 修：forecast_log 也改 fail-closed（讀失敗就 raise/warn 並跳過本次寫入），跟另兩支對齊。

### Y3. 校正「±15pp 不得偏離」的保證，被 normalize 打破
`compute_calibration`（scenario_calibration.py L256–261）先把混合值 clamp 在 rule±15，**再**丟 `_finalize_probs` normalize。normalize 會把數字推出 clamp 邊界。實測：yellow×red 灌 25 筆全 bull → 輸出 `adjusted.bull=42`，但 rule bull=20、上限應是 35 → **實際偏離 +22pp，超過宣稱的 ±15**。
- 影響：單一 bucket 若被一面倒的 realized 灌爆，劇本機率可被推得比規格允許的更偏，護欄失效。
- 修：normalize 之後再 clamp 一次到 rule±15（或在 normalize 前就保留正規化後仍不超界的餘裕）。

### Y4. 校正 bucket 被單一/少數股票的「連日快照」灌爆，n 被高估
`compute_calibration` 依 bucket（技術×籌碼）跨所有股票/日期統計，`n=len(realized_list)`，λ=n/(n+20)。但**同一檔股票連續 20 個交易日的 snapshot 高度自相關**，不是 20 個獨立樣本。現況 2330＋2454 兩檔都落在 `yellow_x_red`，兩檔各記 10 天就湊到 n=20 觸發校正，等於用「兩檔股票的相鄰日」冒充 20 筆獨立勝率去 shrink。單一高流動性股票天天進同一 bucket 就能主導整組校正。
- 修：去重/降權——同 (stock_id) 在同一「劇本窗口」內只算 1 筆，或 n 改成「不重疊的獨立區間數」而非「日快照數」。

### Y5. 持有者 advice「可加碼」無視大盤「禁止新增部位」——與 bull 劇本各說各話
`build_advice` 的 holder 版（primary_decision.py L447–457）在 action=加碼 時照樣輸出「站回均線…可加碼 20 萬」，**完全沒看 `market_new_position`**；`market_banned` 只收斂 nonholder。但同一份大盤閘門，bull 劇本的 `_bull_action`（short_scenarios.py L297–298）遇到「禁止新增部位」會回 wait「不追價僅觀察」。→ **同一個「禁新倉」訊號，劇本層擋、advice 持有者層不擋**。加碼＝新增部位，持有者被告知「可加碼」與戰情室頭條「今天不開新倉」直接打架。
- 現況未觸發（本批無 action=加碼），但結構已在：全綠個股遇大盤紅燈仍可能算出加碼。
- 修：holder 版加碼 plan 也吃 `market_new_position`，禁新倉時降級為「站回訊號明朗前不加碼」，跟 bull 劇本同語意。

### Y6. 短線時間框架 stance 只看技術燈、丟掉籌碼，但 basis 文字明寫「技術＋籌碼」
`primary_decision.build_primary_and_context`（L680）：
```
"short": {"stance": _stance_zh.get(t), "basis": f"技術{...t}＋籌碼{...c}"}
```
stance 只由技術燈 t 派生，**籌碼燈 c 只進 basis 文字、不進 stance**。2330 實例：technical yellow→stance「中性」，但 basis 寫「技術中性＋籌碼偏空」、籌碼實際是 red（連賣 7 天）。使用者讀到「短線 中性（依據：技術中性＋籌碼偏空）」——標籤與依據自相矛盾。舊 `decision_engine.time_frames` 的 short 是 0.6t＋0.4c（會反映籌碼），新派生版退化成只看技術。
- 修：short stance 改用 technical＋chips 合成（沿用舊 short_s 權重），與 basis 一致。

### Y7. 劇本層大盤閘門用 market_light（3 檔），advice/曝險層用 risk_temp（4 檔）——來源不同會分岔
`analyze_tw` 給 short_scenarios 的 `market_new_position_proxy` 是拿 **market_light**（green/amber/red）硬映（L615–617）；而 advice 與 daily 的 `exposure_guidance.new_position` 是拿 **risk_temp** 算（build_exposure_guidance）。兩者來源不同訊號：market_light 紅↔risk_temp≥9 只在極端時對齊，中間帶會分岔（例：market_light green 但 VIX 跳升使 risk_temp=4→「僅限試單」，此時劇本 bull 走「可正常布局→可加碼」、advice 卻「僅限試單」）。註解自稱「門檻精神對齊」，但計算基礎其實不同。
- 修：short_scenarios 的閘門也改吃真正的 `exposure_guidance.new_position`（build 階段已算好，透傳即可），跟 advice 同一個真結果。

### Y8. 三條 log 無鎖，GH Actions 與本機並發會「掉更新」（不會壞檔）
三支 log（recommendation/forecast/scenario）都是 read→append→`_write_json_atomic`（tmp＋os.replace）。os.replace 保證**不會半寫壞檔**，但 read-modify-write 無鎖 → 兩個 build 並發時，後寫者用自己讀到的舊 log 覆蓋，**前者剛 append 的當日 entry 會被吃掉**（lost update，非 corruption）。日常一天一次 GH Actions 影響小，但本機手動重跑撞上排程就會靜默掉資料。
- 修：build 主流程加檔案鎖（flock）或把「同日 append」設計成冪等 merge（讀→以 (date,id) 為鍵 upsert→寫），而非整檔覆蓋。

### Y9.（次要）劇本「base/正常」情境在無更近支撐時，以防守價當支點，把 −8% 下跌畫成預期路徑
2330 現價 2290 與防守 2107 之間沒有其他支撐（MA20/60 在上、MA120 2094 在防守下），故 `_supports` 取到的最近支撐＝防守價本身。base 劇本（守住支撐，40%）於是 price_path＝2290→**回測 2107**→反彈 2324，把「先跌 8% 到防守」當成正常情境的預期路徑；且 base 與 risk 其實共用同一個 2107 關卡（守住/跌破），三劇本塌縮成「2107 這關過不過＋低機率 bull」。
- 修：base 支點優先取「現價附近（±ATR）」的區間中樞，找不到近支撐時 narrative 明講「以現價區間震盪」，不要把遠支撐當回測目標。
