## 2. 兩條 workflow 互相打架風險

- `daily-update.yml`：14:30 台北單次觸發，且有 `concurrency: {group: daily-update, cancel-in-progress: false}`——同一 workflow 重疊時新的會排隊而非互相衝突，這條自己不會跟自己打架。
- `alerts.yml`：09:00-13:30 台北每 15 分一次，**沒有 concurrency 區塊**。
- **時間窗分析**：daily-update（14:30）與 alerts（09:00-13:30）完全不重疊，中間有 1 小時緩衝，正常排程下兩者不會同時跑，**不會互撞**。
- **殘留風險（🟡）**：
  1. alerts 自己連續兩輪重疊——理論上若某輪跑超過 15 分鐘（本次真跑只花 1-2 秒，機率極低，但 FinMind fallback 到即時報價 API 若對方變慢仍有可能），下一輪已經觸發，兩個 runner 同時想 `git push data/alerts_state.json`，其中一個會因為 non-fast-forward 被拒。兩個 workflow 都**沒有 `git pull --rebase` 再 push**的重試機制，push 失敗會讓該 job 直接失敗收場（不影響資料正確性，只是那一輪的「已發送」去重狀態沒存到，最壞情況是同一個 alert 重複發一次 TG 訊息）。
  2. 若 Andy 或未來自動化在交易時段手動 `workflow_dispatch` daily-update，會跟同時段的 alerts 排程撞在一起，push 到不同檔案（`data/`+`public/data/` vs `data/alerts_state.json`）理論上 git 層面可以自動 merge（不同檔案），但如果剛好都在極短時間內 push，仍有極小機率其中一個因為 remote 已前進而被拒（git push 非 merge 式，是要求 fast-forward）。
- **建議**：alerts.yml 補上 `concurrency: {group: alerts, cancel-in-progress: false}`；push 步驟前加一次 `git pull --rebase origin main` 降低非同步失敗率（🟡，非阻塞但值得順手修）。
